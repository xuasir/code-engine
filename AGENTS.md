# 项目规则

## 核心理念
- 简洁至上：遵循 KISS，拒绝过度工程与多余防御。
- 第一性原理：用事实与分析定位问题并提效。
- 尊重代码：严格沿用现有规范与风格。

## 工作流
- 先调研、澄清疑点后再实施。
- 构思方案 → 提请审核 → 拆分任务 → 执行。

## 语言
- 中文为主；术语用英文（API、JWT、Docker 等）。
- 代码元素（变量、函数、路径、CLI）统一英文。
- 机器读字段用英文；人读字段用中文。

## 上下文
1. `contexts/context.md`
2. `README.md`
3. `.agents/specs/*.md`
4. `.agents/workflows/*.md`

## 通用规范
- Implementation Plan 与 Task 标题与说明用中文。
- 新代码注释用中文且简洁。
- 提交信息：`<类型>: <描述>`（如 `feat: 添加登录`）。
- 文档中文，遵循 Markdown。

## 工作模式
- 复杂任务：先读规范，制订计划，完成后更新文档。
- 简单任务：直接实现并保持风格一致。
- 不确定：主动询问并提供选项。

## Spec 驱动开发
- 每次接到需求：
  1) 制定可执行计划
  2) 将计划写入 `.trae/task/plan.md`
  3) 我在对话中审阅并给出“最终计划”
  4) 仅在最终计划确认后开始编码
- 简单任务可直接在对话中审阅，无需写入文件。
